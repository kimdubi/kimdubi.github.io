Psql1

시작하며..

세상엔 수많은 DB들이 있습니다. 이들 각각의 특장점이 무엇인지 파악하고 자신의 서비스에서 어떤 DB가 최고의 퍼포먼스를 낼 수 있는지 파악하는 것은 정말 중요합니다. 이를 위해서는 다양한 DB에 대한 이해가 필요하겠죠?

ORACLE은 강력한 엔터프라이즈 개발 기능과 안정성을, MySQL은 대표적인 오픈소스 DB로 뛰어난 OLTP 성능을 보여주고, CUBRID는 뛰어난 부하 분산 기능을 자랑하는 DB입니다. 그리고 이 글에서 소개해드릴 PostgreSQL은 GIS 지리 정보 처리와 엔터프라이즈급 DBMS의 기능을 제공하는 오픈소스 DB입니다.

이미 북미와 일본쪽에서는 굉장히 인지도가 높은 DB로 자리잡은데 비해 우리나라에서는 GIS 용도로 굳어버린 이미지 때문인지 활용도가 높은 DB는 아닌데요. 그러나 GIS 외에도 기능과 성능면에서 굉장히 뛰어난 DB이기 때문에 알아둘 가치가 충분한 DB입니다. 이번 기회에 그동안 잘 몰랐던 PostgreSQL에 대해 아키텍처부터 시작해서 GIS 까지 소개해드리는 시간을 갖고자 합니다.

PostgreSQL Architecture

먼저 아키텍처입니다! ORACLE 아키텍처에 대해 공부해보신 분이라면 바로 느끼시겠지만 전체적인 아키텍처가 ORACLE과 굉장히 유사합니다. Shared Memory라는 공유메모리가 있고 PostgreSQL 운영에 필요한 백그라운드 프로세스들과 데이터파일들로 구성됩니다.

Shared Memory에서 가장 중요한 요소는 Shared Buffer와 WAL 버퍼입니다. Shared Buffer는 Disk IO를 최소화하기 위해 데이터를 캐싱하는 공간이고 WAL buffer는 이름 그대로 Write Ahead Logging 을 위한 공간으로 데이터베이스의 변경 사항을 WAL 파일로 내리기 전까지 저장해 놓는 공간입니다. PostgreSQL은 데이터를 변경하기 전에 항상 변경 로그부터 기록하는데요 Shared Buffer의 데이터를 변ㄱ여하기 전에 WAL buffer에 로그를 먼저 기록하고, writer process 가 dirty block 을 disk로 내려쓰기 전에 WAL writer process가 WAL logfile로 내려쓰는 작업을 먼저 하는데 이를 Write Ahead Logging 이라고 부릅니다. Shared Memory에서 가장 중요한 공간이니만큼 관련 parameter 설정도 굉장히 중요한데요. 적정 튜닝 값과 파라미터의 의미 또한 글 후반부에서 살펴보겠습니다.

다음은 PostgreSQL의 프로세스 입니다. Shared Memory 위에 보이는 Postmaster process 는 PostgreSQL을 기동할 때 가장 먼저 시작되는 Daemon 프로세스로 DB 기동 시에 recovery 작업, Shared Memory 초기화 작업 그리고 다른 background process 구동 작업을 수행하는 역할을 합니다. 또한 클라이언트 프로세스(JDBC,ODBC등..)의 접속 요청이 있을 때 backend process 를 fork하여 client process와 연결시켜주는 역할을 합니다.

운영에 필요한 PostgreSQL의 백그라운드 프로세스는 아래와 같습니다.

  • writer : 체크포인트 발생 시 dirty 버퍼를 data file로 내려씁니다.
  • WAL writer : WAL buffer 내용을 WAL file에 기록합니다.
  • Checkpointer : 체크포인트 발생 시 dirty 버퍼를 data file에 기록합니다.
  • Archiver : Archive log 모드 일 때 WAL 파일을 지정된 archive 디렉토리에 복사합니다.
  • Logging collector : error 메시지를 log file에 기록합니다.
  • Stats collector : DBMS의 사용 통계정보를 수집하여 세션 정보 (pg_stat_activity), 테이블 통계 정보(pg_stat_all_tables) 같은 정보를 수집합니다.
  • Autovaccum : vacuum이 필요한 시점에 autovacuum worker를 fork하여 vacuum 작업을 수행합니다.

autovacuum 이라는 백그라운드 프로세스를 제외하면 ORACLE의 백그라운드 프로세스와 이름부터 하는 일 까지 거의 동일합니다. autovacuum 백그라운드 프로세스가 수행하는 vacuum이라는 작업은 PostgreSQL의 가장 중요한 내부동작 중 하나인데요. vacuum 을 이해하기 위해서는 PostgreSQL의 MVCC 모델 구현방식을 이해해야 합니다. vacuum에 대해서는 PostgreSQL MVCC 모델에 대해 말씀드리며 간단히 설명드리도록 하겠습니다.

마지막으로 backend process 입니다. Backend process 는 client process 의 쿼리 요청을 수행하고 그 결과를 전송하는 프로세스입니다. 이 때 쿼리 수행에 필요한 메모리가 필요한데 이를 backend process의 local memory라고 부릅니다. Shared Memory와 마찬가지로 적절한 parameter 튜닝을 통해 성능을 최적화하는 것이 중요합니다.

여기까지 어떠신가요? 개인적으로는 ORACLE의 아키텍처와 유사해서 친숙하기도 했고 혹시 차이점은 없읕까 더 궁금하기도 했습니다. 지금부터는 제가 생각하는 ORACLE과 PostgreSQL의 가장 큰 차이점을 살펴보며 PostgreSQL 아키텍처에 대해 조금 더 알아보도록 하겠습니다.

ORACLE과 PostgreSQL 차이점

MVCC 모델 구현

대다수의 RDBMS는 동시성을 위해 MVCC 기능을 제공합니다. MVCC의 기본 원리는 쿼리가 시작된 시점의 트랜잭션ID와 같거나 작은 데이터 버전을 읽는 것 입니다. ORACLE과 MySQL 같은 경우에는 UNDO 세그먼트를 사용하며 쿼리가 시작된 이후에 다른 트랜잭션에 의해 변경된 블록을 만나면 원본 블록으로부터 복사본 블록 (CR copy)를 만들고 그 복사본 블록에 undo segment를 적용하여 쿼리가 시작된 시점으로 되돌려서 읽는 방식을 사용합니다.

반면에 PostgreSQL은 블록 내에 이전 레코드를 저장하는 방식으로 MVCC를 제공합니다. 이 방식은 ORACLE의 undo segment 같은 복잡성 없이 MVCC를 매우 단순하게 구현할 수 있도록 하는 장점이 있습니다. 다만, 이전 데이터와 현재의 데이터를 동일한 테이블 내에 저장하기 때문에 테이블 공간이 비효율적으로 사용될 수 있습니다. 또한 이전의 데이터와 현재의 데이터를 구분하기 위해 시점에 맞는 데이터를 사용하기 위해서는 레코드별로 트랜잭션ID (이하 XID)가 필요한데 PostgreSQL에서는 XID를 위해 4바이트를 사용하고 있어 최대 43억건의 XID를 표현할수 있습니다. 만약 43억건 이후부터 XID 를 1부터 다시 시작하게 되면 어떻게 될까요?

아래의 새로시작된 XID 1은 43억보다 이후의 트랜잭션이지만 XID가 작아서 더 오래된 트랜잭션으로 인식되어 순서가 꼬여버리는 치명적인 문제가 발생할 수 있습니다. 이 두가지 이슈, 불필요한 데이터로 인한 테이블 공간 사용의 비효율성 제거와 오래된 XID 정리를 통해 XID 재사용을 위한 작업이 vacuum입니다. 즉 vacuum은 PostgreSQL 특유의 MVCC 모델에서 비롯될 수 있는 문제점을 해결하기 위해 나온 개념인 것입니다.

Shared Pool의 부재

PostgreSQL은 ORACLE의 Shared Pool 과 같은 기능을 제공하지 않습니다. Shared pool은 수행한 쿼리의 parse tree나 execution plan, 데이터베이스 내 오브젝트 정보를 갖고 있는 공간으로 ORACLE에서는 굉장히 중요한 구성요소 이기 때문에 개인적으로 놀라운 부분이기도 했는데요. PostgreSQL은 공유 메모리 단이 아닌 프로세스 레벨에서 SQL 정보를 공유합니다. 하나의 backend process 내에서 같은 SQL을 여러번 수행하면, 최초 1회만 hard parsing을 수행하게 되는 것 입니다.

마치며

이번 글에서는 PostgreSQL 의 Architecture와 PostgreSQL의 MVCC 구현 방식, vacuum에 대해 간략하게나마 알아보았습니다. 다음 글에서는 DBMS 운영에서 빠질 수 없는 부분, PostgreSQL의 High Availibility 기능에 대해서 알아보도록 하겠습니다.
